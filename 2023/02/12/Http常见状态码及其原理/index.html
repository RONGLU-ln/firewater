HTTP常见的状态码及其原理

状态码是用来反应本次HTTP请求的响应状态，不同的状态有不同的涵义。通过查看HTTP响应报文的状态码，我们可以大致了解这次请求的响应状态。按照不同的响应状态分为以下五类：

1xx

1xx 类状态码属于提示信息，是协议处理中的一种中间状态，还需要后续的操作。

  • 状态码101：状态码101被用来表示协议升级，比如在websocket握手的阶段，客户端会发送http协议通知服务端使用websocket协议通信，服务端如果支持websocket协议，则将响应报文的状态码设置为101。之后双方便可以使用websocket协议进行全双工通信。

上图是一个websocket握手过程，详细了解WebSocket协议可以查看我之前写的 WebSocket握手详解

2xx

2xx类状态码表示本次请求成功，服务器会为我们响应需要的资源

  • 状态码200：最常见的状态，表示本次请求成功，并且在响应报文的body中会返回我们请求的数据。

  • 状态码204：本次请求成功，但是响应报文中只有响应行和响应head没有响应body。

  • 状态码206：本次请求成功，但是响应的body数据只是完整数据的一部分。多用于分块下载技术

    分块下载

    在互联网早期，用户不能像现在这样使用高速宽带访问互联网，那么使用http协议请求一个尺寸稍大的图片或者文件就很吃力了，如果下载过程遇到不可抗拒因素造成网络断开，重新下载又需要重头开始，为了解决这个问题，HTTP1.1引入了分块下载技术，请求头中携带一个range字段，可以指定本次请求资源的范1围，服务器收到后按照请求的范围对相应资源进行分块，然后响应，全部请求完毕后再有客户端对资源进行重组，得到完整的文件。

3xx

3xx类表示重定向，资源位置发生改变，需要重新请求

  • 状态码301：永久重定向，请求的资源已经不在这里了，需要使用新的URL请求资源。

  • 状态码302：临时重定向，请求的资源暂时不在这里，暂时需要使用新的URL请求资源，但一段时间仍使用原本的URL请求资源。

    上述这两个状态我们基本不会察觉到，因为浏览器在收到重定向的状态码后，会直接请求新的URL响应数据，给我们一个直接请求到资源的错觉。

    通过请求模拟工具，我们可以捕捉重定向的响应报文，以我的站点www.799.link为例子，在网站后台的nginx服务器中，443端口监听https://www.799.link 这个路径，请求这个URL会响应数据。80端口监听http://www.799.link 这个路径，收到请求后将请求转发到https://www.799.link 这个URL。

  • 状态码304：缓存重定向，当我们多次请求一个URL时，如果返回状态码304则表示这次请求的数据在服务器并没有修改，直接从本地读取缓存即可。

    如上图，请求这个URL后响应304，通知浏览器从本地读取缓存数据。HTTP缓存机制是HTTP1.1引入的，引入缓存可以提升效率，但是同时也引入了缓存一致性的问题。

缓存重定向

上图中的请求头中携带了Cache-control If-Modified-Since If-None-Match这三个字段，响应头中有ETag Last-Modified这两个字段，HTTP的缓存机制主要使用这个个字段来实现。下面详细介绍一下：

Cache-Control 这个字段用来控制是否使用缓存，可以设置 max-age=n,来表示本地缓存过期时间为n秒，如果没过期则直接从本地读取数据，还可以设置一些值来控制是否使用缓存和谁能缓存文件。

  • no-cache:绕开浏览器换取,请求去询问服务器缓存是否更新,走协商缓存的路线no-store:绕开浏览器缓存和服务器缓存,每次请求都会重新下载完整的资源
  • public:资源可以被所有的用户缓存，包括终端用户和CDN等中间代理服务器private:资源只能被当前客户端缓存,不能被代理服务器缓存
  • private:我们默认的设置,但是通常如果我们设置了max-age就会自动开启public模式

ETag：对文件的一个摘要，只要摘要改变则说明文件发生改变，摘要不改变则说明文件未发生改变。每次服务器响应数据时，也会把响应数据的摘要返回。

Last-Modified：服务器上资源的上一次修改时间，每次服务器响应资源时，也会把上一次修改时间返回。

If-Modified-Since：浏览器请求时会带上上一次请求这个资源响应的Last-Modifie给服务器，让服务器判断这个文件有没有修改

If-None-Match：浏览器请求时会带上上一次请求这个资源响应的ETag给服务器，让服务器判断这个文件有没有修改

有了这些便可实现HTTP缓存，请求过程如下：

  1. 首先通过Cache-Control判断缓存是否过期，没有过期则直接从本地读取，不会发送HTTP请求。这个阶段被称为强制缓存

  2. 如果本地缓存已经过期了，那么则携带着If-Modified-Since 和If-None-Match这两个字段向服务器发送请求，服务器查看请求资源的最后修改时间和资源的摘要是否与请求时携带的If-Modified-Since 和If-None-Match相同，相同则说明资源在服务器上没过期，返回304通知浏览器读取本地缓存，不相同则说明缓存过期了，这时候返回新的数据，并且更新Etag和Last-Modified这两个字段给客户端。这一阶段被称为缓存协商

为什么要有Etag
你可能会觉得使用Last-Modified已经足以让浏览器知道本地的缓存副本是否足够新，为什么还需要Etag呢？HTTP1.1中Etag的出现主要是为了解决几个Last-Modified比较难解决的问题：

一些文件也许会周期性的更改，但是他的内容并不改变(仅仅改变的修改时间)，这个时候我们并不希望客户端认为这个文件被修改了，而重新GET；

某些文件修改非常频繁，比如在秒以下的时间内进行修改，(比方说1s内修改了N次)，If-Modified-Since能检查到的粒度是s级的，这种修改无法判断(或者说UNIX记录MTIME只能精确到秒)；

某些服务器不能精确的得到文件的最后修改时间。

Last-Modified与ETag是可以一起使用的，服务器会优先验证ETag，一致的情况下，才会继续比对Last-Modified，最后才决定是否返回304。

4xx

4xx和5xx都表示错误，不同的是，4xx表示浏览器发生错误，而5xx则说明浏览器没有发生错误，服务器在处理请求时内部发生错误。

常见的状态码如下:

  • 状态码400 ：400 Bad Request，表示请求报文出现错误，比较笼统
  • 状态码403：403 Forbidden 表示请求的资源存在，但是服务器拒绝响应，通常是因为服务器的用户权限不足，所以返回403
  • 状态码404：404 Not Found 表示请求的资源找不到或者不存在，这个状态码还是很出圈的，很多非CS行业的人也知道，比如说唱团体红花会被封杀后将厂牌改名为404，暗示自己被封杀。哈哈哈！很应景！

5xx

5xx 类状态码表示客户端请求报文正确，但是服务器处理时内部发生了错误，属于服务器端的错误码。

  • 状态码500： Internal Server Error与 400 类型，是个笼统通用的错误码。

  • 状态码502： Bad Gateway通常是服务器作为网关或代理时返回的错误码，表示服务器自身工作正常，访问后端服务器发生了错误。

  • 503 Service Unavailable表示当前服务器繁忙无法响应请求，一般是由于后台服务器受到了DDOS攻击或者同时大量人访问服务器，无法及时响应。
